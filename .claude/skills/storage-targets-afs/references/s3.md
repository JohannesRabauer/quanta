# AWS S3 — per-backend reference

## Artifacts

```xml
<dependency>
  <groupId>org.eclipse.store</groupId>
  <artifactId>afs-aws-s3</artifactId>
  <version>${eclipse-store.version}</version>
</dependency>
<dependency>
  <groupId>software.amazon.awssdk</groupId>
  <artifactId>s3</artifactId>
  <version>2.30.11</version>   <!-- or latest -->
</dependency>
```

## General-purpose buckets

```java
S3Client s3 = S3Client.builder()
    .region(Region.EU_NORTH_1)
    .build();   // credentials via default chain

BlobStoreFileSystem fs = BlobStoreFileSystem.New(S3Connector.Caching(s3));
EmbeddedStorage.start(root, fs.ensureDirectoryPath("my-bucket", "data"));
```

Read-after-write consistency: strong since Dec 2020. List-after-write: can be
eventually consistent. Eclipse Store does not rely heavily on LIST for critical
paths, but housekeeping does. Generally fine with caching.

## Directory buckets (S3 Express One Zone)

Lower latency, strongly consistent, zonal. Requires zonal endpoint override.

```java
S3Client s3 = S3Client.builder()
    .region(Region.EU_NORTH_1)
    .endpointOverride(URI.create(
        "https://s3express-eun1-az1.eu-north-1.amazonaws.com"))
    .build();

BlobStoreFileSystem fs = BlobStoreFileSystem.New(S3Connector.CachingDirectory(s3));
```

Recommended for production workloads where latency matters.

## Caching

Always use `S3Connector.Caching(client)` or `S3Connector.CachingDirectory(client)`.
Without caching, every read hits S3 — `start()` alone may do dozens of GET/LIST.

## External configuration

```properties
storage-filesystem.target=aws.s3
storage-directory=bucket-name/folder/subfolder

storage-filesystem.aws.s3.region=eu-north-1
storage-filesystem.aws.s3.endpoint-override=https://...
storage-filesystem.aws.s3.cache=true
storage-filesystem.aws.s3.directory-bucket=false

storage-filesystem.aws.s3.credentials.type=default
```

Use `credentials.type=default` — the AWS SDK default chain (IAM role / env
vars / profile) resolves the right credentials at runtime. The other keys
(`environment-variables`, `system-properties`, `static` plus
`access-key-id` / `secret-access-key`) are listed in `api-catalogue.md` for
reference; do not bake static credentials into the config or the code.

## Credentials in production

Order of preference:

1. **IAM role** on the compute (EC2 instance profile, ECS task role, EKS
   IRSA). `credentials.type=default` picks this up.
2. **Environment variables**: `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`,
   `AWS_SESSION_TOKEN`.
3. **AWS SSO / profile** for developer machines.
4. **Static** — only for controlled environments. Never commit.

## Region and endpoint

- `region` — AWS region (`eu-north-1`, `us-east-1`, …).
- `endpoint-override` — only for S3-compatible services (MinIO, Backblaze) or
  directory-bucket zonal endpoints.

## Cost considerations

- Per-request pricing: aggressive housekeeping ≈ high bill. Tune
  `housekeeping-time-budget` / `housekeeping-interval` to match usage.
- Data transfer: cross-region reads are expensive. Keep storage in the same
  region as the compute.
- Directory buckets have different pricing (higher per-GB, lower per-request).
  Worthwhile for latency-sensitive apps.

## Backup to S3 from local

```properties
storage-directory=data                                # local NIO
backup-filesystem.target=aws.s3
backup-directory=my-bucket/backup
backup-filesystem.aws.s3.region=eu-north-1
backup-filesystem.aws.s3.credentials.type=default
```

## S3 versioning / object lock

Orthogonal to Eclipse Store. Enable on the bucket for disaster-recovery safety.
Costs extra.

## Gotchas

- **Retention policies** on the bucket may block housekeeping deletes — the
  bucket will accept the delete but keep older object versions. Not harmful but
  costs extra.
- **Request rate limits**: S3 scales request rates per prefix. Single-prefix
  directories (the default Eclipse Store layout) can hit limits with very high
  write rates. Use multiple channels to spread across prefixes.
- **Eventual consistency on LIST in legacy regions**: rare now, but be aware.
- **Concurrent writers**: still a one-writer-per-bucket rule at the Eclipse Store
  level. Two managers against the same bucket/prefix is invalid.
